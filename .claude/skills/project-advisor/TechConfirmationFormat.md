# 技術者確認フォーマット

本ファイルは、営業ヒアリングまとめ生成後に技術部門へ送付する確認フォーマットを定義する。
営業担当がコピペしてSlack・メール等で技術者に送れる形式とする。

---

## フォーマット本体

以下をそのまま技術担当者に送付してください。

---

**【技術確認依頼】{顧客名} {案件名}**

お疲れ様です。以下の案件について提案可否の検討をしています。
技術面から4点確認させてください。各項目、最も近い選択肢を選んでいただければ大丈夫です。

### 案件概要

- **顧客**: {顧客名}（SB様経由）
- **案件名**: {案件名}
- **概要**: {案件概要を2-3行で}
- **規模**: {概算金額・期間}
- **主な技術要件**: {ヒアリングで得た技術要件}

---

### 確認1: 技術的学習効果（当部にとっての技術的な学び）

営業担当の見立てでは「{営業の回答}」とのことでした。技術視点でどう見えますか？

以下から最も近いものを選んでください:

- [ ] **(A)** 当部にとって初の領域で、高い学習価値がある
- [ ] **(B)** 新しい技術要素があり、一定の学習効果がある
- [ ] **(C)** 既存スキルの延長だが、大規模適用等で経験値は得られる
- [ ] **(D)** 既知の領域で新規性が乏しい
- [ ] **(E)** 既存スキルで完全にカバー済み、新規性なし

補足コメント（任意）:

---

### 確認2: 要件適合度（当社のソリューション/パッケージとのフィット感）

以下の観点を参考に判断してください:
- 類似案件の実績はあるか（過去に同じ業界・同じ技術スタックで納品した経験）
- 必要な技術スタック（言語・FW・ミドルウェア等）に対する社内の知見・経験
- 既存資産（パッケージ・テンプレート・共通基盤等）の流用可能性
- カスタマイズが発生する場合の工数規模感（人月単位の概算イメージ）

上記を踏まえ、最も近いものを選んでください:

- [ ] **(A)** 既存パッケージ/ソリューションで完全にカバーできる
- [ ] **(B)** 既存で8割カバー、軽微なカスタマイズで対応可能
- [ ] **(C)** 一部カスタマイズが必要（中程度の開発が発生）
- [ ] **(D)** 大幅なカスタマイズまたは新規開発が必要
- [ ] **(E)** 技術的にフィットしない

補足コメント（任意）:

---

### 確認3: セキュリティ/コンプライアンス適合

以下から最も近いものを選んでください:

- [ ] **(A)** 当社の標準セキュリティ基準で完全に適合
- [ ] **(B)** 概ね適合、軽微な対応で充足
- [ ] **(C)** 方針レベルで検討が必要
- [ ] **(D)** 追加の認証取得やコンプライアンス対応が必要
- [ ] **(E)** 重大なセキュリティ/コンプライアンスリスクあり

補足コメント（任意）:

---

### 確認4: リソース確保可能性

以下から最も近いものを選んでください:

- [ ] **(A)** 当部に経験者がいて、専任で即確保可能
- [ ] **(B)** 他部署に経験者がいて、連携が実際に進んでおり、専任確保の見込みあり
- [ ] **(C)** 他部署に経験者がいるが連携未確定、または外部調達で確保可能
- [ ] **(D)** 確保の見通しが立っていない（連携不可、部分確保のみ等）
- [ ] **(E)** 確保不可能

補足コメント（任意）:

### 確認5: コスト試算（プロジェクトプロファイル実施時のみ）

営業担当が収集した概算費用感をもとに、技術視点で試算をご確認ください。

営業の概算: {営業がPhase 5で回答したコスト感触}

以下から最も近いものを選んでください:

- [ ] **(A)** 概算費用の試算が完了しており、利益率30%以上が見込める
- [ ] **(B)** 概算費用のめどが立っており、利益率20〜30%が見込める
- [ ] **(C)** 試算は可能だが、利益率は標準的（目安として20%前後）
- [ ] **(D)** 費用感はあるが試算の根拠が弱く、利益率が不明確
- [ ] **(E)** コスト情報が不十分で、費用超過リスクが懸念される

- [ ] **(B)** 既存製品で概ねカバー可能。軽微なカスタマイズが必要だが問題なし
