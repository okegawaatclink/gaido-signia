# スコアリング基準

本ファイルは案件スコアリングの判定チェーン・ウェイト・閾値・2Step判定マトリクスを定義する。
スコアリング時は必ず本ファイルの基準に従い、各項目で `evidence → chain_match → score` の3段階を記録すること。

## 基本ルール

- **曖昧時は低い方のスコアを採用**する（楽観バイアスの排除）
- **「存在する」と「活用できる」は異なる**。他部署にリソースがいても連携が取れなければスコアは下がる
- **部分的な確保（0.5人月等）はスコア2**（コンテキストスイッチ・責任の曖昧さにより「確保困難」扱い）
- SB様経由の情報はすべて伝聞。直接確認できていない場合は1段階下げる

---

## 1. ゲート項目（最低条件）

いずれか1つでもスコア3未満 → **ゲートNG**（早期判定）

### 1.1 予算確度（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 予算が正式に承認されている（稟議番号あり等） |
| 4 | 予算が申請済みで承認見込みが高い |
| 3 | 予算が申請中または確保の方向で調整中 |
| 2 | 予算は検討中（未申請） |
| 1 | 予算状況が不明 |

### 1.2 意思決定者アクセス（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | SBが意思決定者と直接やり取りしている |
| 4 | SBが意思決定者にアクセスできる（間接的） |
| 3 | SBが間接的に情報を得ている |
| 2 | 意思決定者の存在は聞いているがアクセス手段が不明 |
| 1 | 意思決定者が誰か不明 |

### 1.3 推進者の有無（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 影響力のある人が推進者として確定し、積極的に動いている |
| 4 | 影響力のある人が関与しているが、推進者として確定かは曖昧 |
| 3 | 推進者の候補はいるが確定ではない |
| 2 | 推進者がいるかもしれないが具体的な情報がない |
| 1 | 推進者が誰か不明 |

## 2. 確度軸（営業主体で評価）

### 2.1 競合有無・優位性（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 競合なし、または指名案件 |
| 4 | 競合はいるが当社が優位（技術力・実績等で差別化） |
| 3 | 競合と互角 |
| 2 | 競合が優位だが逆転の余地あり |
| 1 | 競合が圧倒的に優位 |

### 2.2 商流の複雑度（weight: 4）

※CTCから見て、エンドユーザーまでの間に何社入るか。間の会社が多いほど情報の正確性が下がり、調整コストが増える。

| スコア | 条件 |
|--------|------|
| 5 | SB様との直接取引（エンド→SB→CTC） |
| 4 | SB様＋仲介1社（エンド→代理店→SB→CTC） |
| 3 | 仲介2社以上だが各社の役割・連絡先が明確 |
| 2 | 仲介2社以上で関係が複雑、または情報伝達に遅延・ロスが発生 |
| 1 | 商流が不明確（誰がどの立場か把握できていない） |

### 2.3 導入期限の強さ（weight: 4）

| スコア | 条件 |
|--------|------|
| 5 | 法令対応や外的圧力による必須期限あり |
| 4 | 明確な期限あり（年度内導入等） |
| 3 | 6ヶ月以内に導入したい意向 |
| 2 | 1年以内にはやりたいが緊急性なし |
| 1 | 期限未定・情報なし |

---

## 3. 戦略軸（営業＆技術で共に評価）

### 3.1 戦略整合性（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 当部の重点領域のど真ん中。関連部署との連携も取れる状態 |
| 4 | 重点領域に近い。連携体制も概ね整っている |
| 3 | 一定の関連はあるが、直接の重点ではない |
| 2 | 関連部署と連携が取れる状態ではない、または当部の重点から外れる |
| 1 | 重点顧客外かつ業界外 |

※「会社としての重点」と「当部の重点」を区別すること。他本部の重点顧客であっても、当部との連携が取れない場合はスコアを下げる。

### 3.2 事例としての活用可能性（weight: 3）

※事例化の目的: 「CTCが○○社のプロジェクトを手がけた」と紹介できることで、同業界・同規模の新規顧客への提案時に大きなアドバンテージとなる。

| スコア | 条件 |
|--------|------|
| 5 | CTC名を出して事例公開が確定している |
| 4 | 顧客が事例公開に前向き（CTC名あり） |
| 3 | 条件次第では事例化可能（顧客と未協議だが可能性あり） |
| 2 | SB様の事例としては公開されるが、CTCの関与は非公開。社内ナレッジや匿名事例としてのみ活用可能 |
| 1 | NDA等の制約で事例化不可（関与自体を外部に言えない） |

※CTC名を出せない場合でも、社内の提案ナレッジとしては価値があるため、最低スコア（1）にはしない。

### 3.3 技術的学習効果/PoC価値（weight: 4）

| スコア | 条件 |
|--------|------|
| 5 | 当部にとって初の領域で、高い学習価値がある |
| 4 | 新しい技術要素があり、一定の学習効果 |
| 3 | 既存スキルの延長だが、大規模適用等で経験値 |
| 2 | 既知の領域で新規性が乏しい |
| 1 | 既存スキルで完全にカバー済み、新規性なし |

### 3.4 ブランド価値/市場影響（weight: 3）

| スコア | 条件 |
|--------|------|
| 5 | 業界トップ企業で高い知名度 |
| 4 | 大手企業で一定の知名度 |
| 3 | 中堅企業（中立） |
| 2 | 小規模企業 |
| 1 | ブランド価値なし |

### 3.5 重点顧客/産業への寄与（weight: 4）

| スコア | 条件 |
|--------|------|
| 5 | 重点産業に直結し、関係強化に大きく貢献 |
| 4 | 重点産業の関係強化に一定の寄与 |
| 3 | 重点産業に一定の関連はある |
| 2 | 重点産業との関連が薄い |
| 1 | 重点顧客外・重点産業外 |

### 3.6 将来パイプライン拡張性（weight: 4）

| スコア | 条件 |
|--------|------|
| 5 | 横展開・追加案件の可能性が非常に高い |
| 4 | 横展開の可能性が高い |
| 3 | 一定の拡張可能性がある |
| 2 | 拡張の見込みは薄い |
| 1 | 単発案件で拡張性なし |

## 4. 提案難易度軸（技術主体で評価）

### 4.1 要件適合度/フィットギャップ（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 既存パッケージ/ソリューションで完全にカバー |
| 4 | 既存で8割カバー、軽微なカスタマイズで対応 |
| 3 | 一部カスタマイズが必要（中程度の開発） |
| 2 | 大幅なカスタマイズまたは新規開発が必要 |
| 1 | 技術的にフィットしない |

### 4.2 セキュリティ/コンプライアンス適合（weight: 5）

| スコア | 条件 |
|--------|------|
| 5 | 当社の標準セキュリティ基準で完全に適合 |
| 4 | 概ね適合、軽微な対応で充足 |
| 3 | 方針レベルで検討が必要 |
| 2 | 追加の認証取得やコンプライアンス対応が必要 |
| 1 | 重大なセキュリティ/コンプライアンスリスクあり |

### 4.3 リソース確保可能性（weight: 4）

| スコア | 条件 |
|--------|------|
| 5 | 当部に経験者がいて、専任で即確保可能 |
| 4 | 他部署に経験者がいて、連携が実際に進んでおり、専任確保見込み |
| 3 | 他部署に経験者がいるが連携未確定、または外部調達で確保可能 |
| 2 | 確保見通しが立っていない（連携不可、0.5人月等の部分確保） |
| 1 | 確保不可能 |

※「リソースが存在する」と「リソースを活用できる」は異なる。他部署の経験者がいても連携が取れなければスコアは下がる。
※部分的な確保（0.5人月等）はコンテキストスイッチ・責任の曖昧さにより「確保困難」（スコア2）。

## 5. 社内共有準備軸（営業＆技術で共に評価）

※Phase 5「社内共有準備フェーズ」で評価する。営業まとめ後に「社内共有準備をする」を選択した場合のみ実施。

### 5.1 コスト軸（weight: 4）

社内説明に必要なコスト情報の充足度と収益性見込みを評価する。

| スコア | 条件 |
|--------|------|
| 5 | 概算費用の試算が完了しており、利益率30%以上が見込める |
| 4 | 概算費用のめどが立っており、利益率20〜30%が見込める |
| 3 | 概算費用の試算が可能な情報は揃っており、利益率はおおむね標準的 |
| 2 | 費用感はあるが試算の根拠が弱く、利益率が不明確 |
| 1 | コスト情報がほぼなく、費用超過リスクが懸念される |

### 5.2 製品軸（weight: 3）

提案する製品・ソリューションの適合度とライセンス・サポート体制の明確さを評価する。

| スコア | 条件 |
|--------|------|
| 5 | 既存製品でほぼカバー可能で、ライセンスとサポート体制が明確 |
| 4 | 既存製品で概ねカバー可能。軽微なカスタマイズが必要だが問題なし |
| 3 | パートナー製品との組み合わせが必要。ライセンス・サポートは概ね見通しあり |
| 2 | 新規製品の調達や大幅なカスタマイズが必要。ライセンスや保守体制に不確実性あり |
| 1 | 自社・パートナー製品では対応困難、または適切な製品の見通しが立っていない |

### 5.3 リスク軸（weight: 5）

プロジェクト遂行上の総合リスクを評価する。プロジェクト・技術・契約の3観点を総合的に判断する。

| スコア | 条件 |
|--------|------|
| 5 | リスクは限定的で、すべて対応策が明確 |
| 4 | 一定のリスクはあるが管理可能。対応策の見通しがある |
| 3 | 複数のリスク要因があるが、許容範囲内で対処方針は立てられる |
| 2 | 大きなリスクがあり、対応策が不明確。上長への相談が必要な水準 |
| 1 | 受け入れ難いリスクがある（契約条件の問題、技術的不確実性が高い、組織体制の問題等） |

**リスク観点の例**:
- プロジェクトリスク: スコープ不明確、顧客側体制の不安定さ、期間の非現実性
- 技術リスク: 未実証技術の採用、技術的負債、依存する外部サービスの不安定さ
- 契約リスク: 不利なSLA・ペナルティ条項、知財の帰属問題、秘密保持の過剰な制約

## 6. ランク判定

各軸の達成率を算出し、ランクを判定する。

**算出方法**: 達成率 = 加重合計スコア / 加重合計最大スコア

| 達成率 | ランク |
|--------|--------|
| 0.7 以上 | High |
| 0.5 以上 0.7 未満 | Mid |
| 0.5 未満 | Low |

**各軸の最大スコア（加重）**:
- 確度軸: 5×5 + 4×5 + 4×5 = 65
- 戦略軸: 5×5 + 3×5 + 4×5 + 3×5 + 4×5 + 4×5 = 115
- 提案難易度軸: 5×5 + 5×5 + 4×5 = 70
- 社内準備度（コスト・製品・リスク合算）: 4×5 + 3×5 + 5×5 = 60

※提案難易度のランクは **逆転**する（Low=提案が易しい、High=提案が難しい）

## 7. 3Step判定マトリクス

### Step 1: (戦略 × 確度) → 案件魅力度

| 戦略 ＼ 確度 | High | Mid | Low |
|-------------|------|-----|-----|
| **High** | Go（Aチーム投入） | Conditional Go | No Bid |
| **Mid** | Proceed（標準体制） | Conditional | No Bid |
| **Low** | Proceed（限定） | Deprioritize | No Bid |

### Step 2: (案件魅力度 × 提案難易度) → 暫定アクション

| 魅力度 ＼ 提案難易度 | Low（易） | Mid | High（難） |
|--------------------|-----------|-----|-----------|
| **Go系** | 即実行/Aチーム投入 | 体制強化して実行 | 上長に相談 |
| **Conditional系** | 標準体制で実行 | 条件付き継続 | 見送り |
| **Proceed（標準）** | 標準体制で実行 | 条件付き継続 | 見送り |
| **Proceed（限定）** | 余力があれば対応 | 見送り | 見送り |
| **Deprioritize** | 余力があれば対応 | 見送り | 見送り |
| **No Bid系** | 見送り | 見送り | 見送り |

### Step 3: (暫定アクション × 社内準備度) → 最終アクション

※Phase 5を実施した場合のみ適用する。Phase 5未実施の場合はStep 2の結果がそのまま最終アクションになる。

| 暫定アクション ＼ 社内準備度 | High | Mid | Low |
|--------------------------|------|-----|-----|
| **即実行/Aチーム投入** | 即実行（社内準備OK） | 即実行（コスト/リスク情報補完要） | 上長に相談 |
| **体制強化して実行** | 体制強化して実行 | 体制強化して実行（情報補完要） | 条件付き継続 |
| **上長に相談** | 上長に相談 | 上長に相談（情報補完要） | 見送り |
| **標準体制で実行** | 標準体制で実行 | 標準体制で実行（情報補完要） | 条件付き継続 |
| **条件付き継続** | 条件付き継続 | 条件付き継続（情報補完要） | 見送り |
| **余力があれば対応** | 余力があれば対応 | 余力があれば対応 | 見送り |
| **見送り系** | 見送り | 見送り | 見送り |
