# 技術調査・比較手順

## 概要

Epicの実現方法について、技術的な調査と比較検討を行う手順です。

## 実施タイミング

以下の場合に技術調査を実施:

- 複数の実現方法が考えられる
- 最新の技術動向を確認したい
- ユーザーが技術選定で迷っている
- 実現可能性の確認が必要

## 調査手順

### 1. 調査観点の明確化

ユーザーと以下を確認:

- 何を比較したいか（ライブラリ、アプローチ、アーキテクチャ等）
- 重視する評価軸（精度、コスト、開発速度、保守性等）
- 制約条件（既存システム、予算、スキルセット等）

### 2. Web検索による情報収集

**WebSearchツール** を使用して最新情報を収集:

```
検索クエリ例:
- "[技術A] vs [技術B] comparison 2025"
- "[課題] best practices 2025"
- "[ライブラリ名] pros cons"
```

**注意:**
- 検索結果のURLを必ず記録
- 公式ドキュメント、GitHub、技術ブログを優先
- 古い情報（2年以上前）は参考程度に

### 3. 比較表の作成

収集した情報を比較表にまとめる:

```markdown
| 観点 | 選択肢A | 選択肢B | 選択肢C |
|-----|--------|--------|--------|
| 概要 | ... | ... | ... |
| メリット | ... | ... | ... |
| デメリット | ... | ... | ... |
| [評価軸1] | ◎/○/△/× | ... | ... |
| [評価軸2] | ... | ... | ... |
| GAiDo組み込み | ... | ... | ... |
```

### 4. ユーザーとの対話

比較表を提示し、以下を確認:

- 追加で調査すべき観点はあるか
- 評価軸の重み付けはどうか
- 懸念点や質問はあるか

ユーザーの質問に応じて追加調査を実施。

### 5. 推奨アプローチの提示

調査結果をもとに推奨アプローチを提示:

```markdown
## 推奨: [選択肢X]

**理由:**
1. [理由1]
2. [理由2]
3. [理由3]

**検証が必要な点:**
- [不確実な点1]
- [不確実な点2]
```

### 6. Spike要否の判断

以下に該当する場合、Spikeを提案:

| 状況 | Spike要否 |
|-----|----------|
| 実現可能性が不明 | 必要 |
| 複数アプローチで精度比較が必要 | 必要 |
| 見積もりに必要な情報が不足 | 必要 |
| 技術選定が確定している | 不要 |
| 既存の知見で十分 | 不要 |

## 出力

技術調査の結論を以下の形式でまとめ、Epic作成時に引用:

```markdown
## 技術検討結論

### 採用アプローチ
[選択した方式の概要]

### 選定理由
1. [理由1]
2. [理由2]

### アーキテクチャ概要
```
[図やフローの説明]
```

### 参考URL
- [タイトル1](URL1)
- [タイトル2](URL2)
```

## 注意事項

- 必ず根拠となるURLを記載すること
- ハルシネーション防止のため、不明点は「要検証」と明記
- ユーザーの意見を尊重し、押し付けないこと
