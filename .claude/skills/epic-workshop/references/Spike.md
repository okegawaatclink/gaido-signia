# Spike Issue作成手順

## 概要

技術検証が必要な場合に、Spike Issueを作成する手順です。
SpikeはEpicの前提として作成し、検証完了後にEpicの実装に進みます。

## Spikeとは

- 技術的な不確実性を解消するための調査・検証タスク
- 実装の見積もりや方針決定に必要な情報を得る
- 成果物は「検証結果レポート」と「推奨アプローチの決定」

## 作成タイミング

以下の場合にSpike Issueを作成:

- 実現可能性が不明確
- 複数のアプローチを実際に試して比較したい
- 外部サービス/ライブラリの動作検証が必要
- パフォーマンスや精度の実測が必要

## テンプレート

```markdown
## 概要
[何を検証するか、なぜ検証が必要かを簡潔に記載]

## 検証項目

### 1. [検証カテゴリ1]
- [ ] [検証項目1-1]
- [ ] [検証項目1-2]
- [ ] [検証項目1-3]

### 2. [検証カテゴリ2]
- [ ] [検証項目2-1]
- [ ] [検証項目2-2]

### 3. [検証カテゴリ3（比較検証の場合）]

| 比較項目 | 選択肢A | 選択肢B |
|---------|--------|--------|
| [観点1] | 要検証 | 要検証 |
| [観点2] | 要検証 | 要検証 |

- [ ] [選択肢A]の検証: [検証内容]
- [ ] [選択肢B]の検証: [検証内容]
- [ ] 両方式で同一条件で比較し、品質評価

### 4. 評価基準
- [評価基準1]
- [評価基準2]
- [評価基準3]

## 成果物
- 各方式のサンプル出力（同一内容で比較）
- 比較評価レポート
- 推奨アプローチの決定
- 採用方式の実装サンプル
```

## Issue作成方法

**必ず `tools/create_issues.py` を使用してIssueを作成すること。** `gh issue create` を直接使用してはならない。

### 手順

1. Issue定義をJSONファイルに書き出す:

```json
[
  {
    "title": "[Spike] [検証内容の簡潔な説明]",
    "body": "[テンプレートに従った本文]",
    "labels": ["Spike"],
    "issueType": "Spike"
  }
]
```

2. `tools/create_issues.py` で登録する:

```bash
python3 tools/create_issues.py \
  https://github.com/[owner]/[repo] \
  [JSONファイルパス] --yes
```

**注意:** リポジトリURLは `gh repo view --json url -q .url` で取得できる。

## タイトル命名規則

```
[Spike] [何を][どうする]技術検証
```

例:
- `[Spike] RFPからPowerPoint提案書を自動生成する技術検証`
- `[Spike] WebSocket vs SSEのリアルタイム通信方式比較`
- `[Spike] Claude APIのストリーミングレスポンス実装検証`

## 検証項目の書き方

### 良い例
```markdown
- [ ] PDF形式のRFP読み込み（PyMuPDF/pdfplumber）
- [ ] Markdown→PPTX変換精度、テンプレート適用、画像埋め込み
```

### 悪い例
```markdown
- [ ] PDFを読む
- [ ] PowerPointを作る
```

**ポイント:**
- 具体的なツール/ライブラリ名を含める
- 検証の観点（精度、速度、etc）を明記
- チェック可能な粒度にする

## 注意事項

- Spikeは「調査」であり「実装」ではない
- タイムボックスを意識（1-3日程度が目安）
- 検証結果は必ずIssueコメントまたは別ドキュメントに記録
- 検証完了後、Epicの実装可否・方針を決定

## 完了後

Spike Issue作成後、続けてEpic Issueを作成する。
EpicのBody内で `前提Spike: #[Spike Issue番号]` として参照する。
