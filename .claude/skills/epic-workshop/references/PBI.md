# PBI Issue作成手順（対話ベース版）

## 概要

対話で練り上げた内容をもとに、PBI Issueを作成する手順です。
技術検討の結論やSpikeへの参照を含めた、充実したPBIを作成します。

## 前提条件

- ユーザーとの対話で背景・課題が明確になっていること
- 技術検討が必要な場合は、`TechResearch.md` の手順を完了していること
- Spikeが必要な場合は、`Spike.md` の手順でSpike Issueを作成済みであること

## テンプレート

```markdown
## 背景
[課題・ペインを箇条書きで記載]
- [課題1]
- [課題2]
- [課題3]

## ゴール
[このPBIで達成したいことを1-2文で記載]

## 想定フロー
1. **[ステップ1]**: [説明]
2. **[ステップ2]**: [説明]
3. **[ステップ3]**: [説明]
...

## アーキテクチャ
```
[フロー図やアーキテクチャをテキストまたはMermaidで記載]
```

## 含まれる機能（候補）
- [機能1]
- [機能2]
- [機能3]

## 前提Spike
- #[Spike Issue番号] [Spikeタイトル]

（Spikeがない場合はこのセクションを省略）

## 受け入れ条件
- [ ] [条件1]
- [ ] [条件2]
- [ ] [条件3]
```

## タイトル命名規則

ユーザーストーリー形式:

```
[ユーザー種別]は、[機能/改善]をしたい。なぜなら[課題/ペイン]だからだ
```

例:
- `利用者は、GAiDoで提案書を作りたい。なぜなら提案書の構成・内容を作成するのに非常に労力がかかり時間がないからだ`
- `開発者は、MacでGAiDoを稼働させたい。なぜなら開発時に複数のコンテナを起動し動作確認を行うことが多いからだ`

## Issue作成方法

**必ず `tools/create_issues.py` を使用してIssueを作成すること。** `gh issue create` を直接使用してはならない。

### 手順

1. Issue定義をJSONファイルに書き出す:

```json
[
  {
    "title": "[ユーザーストーリー形式のタイトル]",
    "body": "[テンプレートに従った本文]",
    "labels": ["PBI"],
    "issueType": "PBI"
  }
]
```

2. `tools/create_issues.py` で登録する:

```bash
python3 tools/create_issues.py \
  https://github.com/[owner]/[repo] \
  [JSONファイルパス] --yes
```

**注意:** リポジトリURLは `gh repo view --json url -q .url` で取得できる。

## 各セクションの書き方

### 背景
- ユーザーとの対話で確認した課題・ペインを記載
- 具体的な状況を含める（「〜に時間がかかる」「〜が属人的」など）

### ゴール
- 1-2文で端的に
- 「〜できるようになる」形式を推奨

### 想定フロー
- ユーザー視点での操作フローを記載
- 番号付きリストで段階を明確に
- 各ステップは太字で見出しを付ける

### アーキテクチャ
- 技術検討の結論を反映
- テキストベースのフロー図またはMermaid記法
- コンポーネント間の関係を明確に

### 含まれる機能（候補）
- 「候補」として柔軟性を持たせる
- 実装時の参考になるレベルで記載

### 前提Spike
- Spikeがある場合は必ず記載
- Issue番号でリンクを作成（`#123` 形式）

### 受け入れ条件
- チェックボックス形式
- テスト可能な具体的な条件
- ユーザー視点での完了条件

## 注意事項

- 対話で確認した内容を正確に反映すること
- 作成後、ユーザーにIssue URLを報告すること

## 完了後

PBI Issue作成後、以下を報告:

```markdown
## 作成完了

| Issue | タイトル | URL |
|-------|---------|-----|
| Spike | [タイトル] | [URL] |
| PBI | [タイトル] | [URL] |

**次のアクション:**
- Spikeがある場合: Spike検証を実施後、実装着手へ
- Spikeがない場合: 実装着手へ
```
