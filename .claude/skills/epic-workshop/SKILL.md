# Epic Workshop

## 概要

ユーザーとの対話を通じて、PBI（と必要に応じてSpike）を練り上げ、GitHub Issueとして作成するスキルです。

**既存の `/backlog` との違い:**
- `/backlog`: 確定済み要件からPBI/Taskを作成
- `/epic-workshop`: 対話で要件を練り上げながらPBI+Spikeを作成

## 使用場面

- 新しい機能のアイデアがあるが、詳細が固まっていない
- 技術検証（Spike）が必要かどうか判断したい
- 技術的なアプローチを比較検討してからPBIを作りたい

## フロー

```mermaid
flowchart TD
    A[/epic-workshop 開始] --> B[背景・課題のヒアリング]
    B --> C[ユーザーストーリー形式で整理]
    C --> D{技術検討が必要?}
    D -->|YES| E[技術調査・比較]
    E --> F[ユーザーと対話で検討]
    F --> G{Spikeが必要?}
    G -->|YES| H[Spike Issue作成]
    H --> I[PBI Issue作成]
    G -->|NO| I
    D -->|NO| I
    I --> J[完了報告]
```

## 実行手順

### Step 1: 背景・課題のヒアリング

ユーザーに以下を確認する:

1. **誰が**: このPBIの対象ユーザーは誰か
2. **何をしたい**: どんな機能・改善を求めているか
3. **なぜ**: なぜその機能が必要か（課題・ペイン）
4. **補足情報**: 制約、期限、技術的な前提など

**質問方式**:
- 各質問は **AskUserQuestion** で4択の選択肢UIを提示すること
- AIがユーザーの依頼内容や文脈に応じて、適切な4つの選択肢を生成する
- 選択肢の最後には「その他（自由記述）」を含め、選択肢に該当しない場合にも対応する
- 関連する質問はまとめて提示してよいが、1回のAskUserQuestionで最大4問まで
- **重要**: `questions` パラメータは必ず配列として渡すこと。JSON文字列は禁止

**AskUserQuestion呼び出し例**:
```
AskUserQuestion(
  questions=[
    {
      "question": "このPBIの対象ユーザーは誰ですか？",
      "header": "対象ユーザー",
      "multiSelect": false,
      "options": [
        {"label": "エンドユーザー", "description": "サービスを利用する一般ユーザー"},
        {"label": "管理者", "description": "システム管理・運用を行う管理者"},
        {"label": "開発者", "description": "APIやSDKを利用する開発者"},
        {"label": "その他", "description": "上記以外（自由記述で補足してください）"}
      ]
    }
  ]
)
```

**注意**: 上記は例示。実際の選択肢はユーザーの依頼内容に応じて毎回AIが生成すること。

### Step 2: ユーザーストーリー形式で整理

ヒアリング内容を以下の形式で整理し、ユーザーに確認:

```
[ユーザー種別]は、[機能/改善]をしたい。
なぜなら[課題/ペイン]だからだ。
```

### Step 3: 技術検討（必要な場合）

**Readツールで `references/TechResearch.md` を読み込み**、手順に従って技術調査を実施。

- Web検索で最新情報を収集
- 複数のアプローチを比較表で整理
- メリット・デメリットを明確化
- ユーザーと対話しながら方針を決定

### Step 4: Spike作成（必要な場合）

以下の条件に該当する場合、Spikeを先に作成:

- 技術的な実現可能性が不明
- 複数のアプローチがあり、検証が必要
- 見積もりに必要な情報が不足

**Readツールで `references/Spike.md` を読み込み**、手順に従ってSpike Issueを作成。

### Step 5: PBI作成

**Readツールで `references/PBI.md` を読み込み**、手順に従ってPBI Issueを作成。

- Spikeがある場合は、PBIの「前提Spike」として参照
- 技術検討の結論を「アーキテクチャ」セクションに反映

### Step 6: 完了報告

作成したIssueの一覧をユーザーに報告:

- Spike Issue（あれば）: 番号、タイトル、URL
- PBI Issue: 番号、タイトル、URL
- 次のアクション（Spike検証 or 実装着手）

## 注意事項

- 対話を重視し、ユーザーの意図を正確に把握すること
- 技術検討では根拠（URL、ドキュメント）を示すこと
- Spikeの要否はユーザーと相談して決定すること
- 作成したIssue番号は必ず報告すること
