# 実装の振り返り（problems.md作成）手順

## 概要

今回の実装で苦労した点を振り返り、`problems.md` としてプロジェクトルートに出力する。

## 振り返りの観点

以下の観点で、実装中に苦労した点・時間を浪費した点を洗い出す:

- **ルール不足**: CLAUDE.mdやrules/に記載がなく、試行錯誤が必要だった
- **やり直し**: 一度作ったものを作り直す必要があった
- **環境構築**: Docker、依存関係、ポート設定等で詰まった
- **context消失**: コンテキストウィンドウの圧縮で情報が失われた
- **手戻り**: テストで問題が見つかり、実装を修正した

## problems.mdテンプレート

```markdown

| 項目 | 値 |
|------|-----|
| プロジェクト名 | [プロジェクト名] |
| 記録項目数 | N件 |
| 記録日 | YYYY-MM-DD |

## 一覧

| # | 項目名 | 苦労度 | 深刻度 | 原因カテゴリ | 影響範囲 |
|---|--------|--------|--------|-------------|---------|
| 1 | [項目名] | ★★★☆☆ | ★★☆☆☆ | ルール不足 | ルール改善で予防可能 |
| 2 | [項目名] | ★★★★☆ | ★★★★☆ | 環境問題 | Output System固有 |
| ... | ... | ... | ... | ... | ... |

## 各項目の詳細

### 1. [項目名]

- **苦労度**: ★★★☆☆（3/5）
- **深刻度**: ★★☆☆☆（2/5）
- **原因カテゴリ**: ルール不足
- **影響範囲**: ルール改善で予防可能

#### 何が起きたか

[具体的に何が起きたかを記述。エラーメッセージがあればそのまま引用する]

#### 原因

[なぜそれが起きたのかを記述。根本原因まで掘り下げる]

#### どう解決したか

[実際にどう解決したかを記述。変更したファイル名・変更内容を含める]

#### 改善提案

以下の形式で、**別のコンテキストのAIエージェントが読んで即座に実装できる**レベルで記述する。

| 項目 | 内容 |
|------|------|
| 対象ファイル | [変更すべきファイルの相対パス] |
| 変更種別 | [新規追加 / 既存修正 / 既存削除] |

**具体的な変更内容:**

[追加・修正すべき内容をそのまま書く。ルール追加ならルール文面、テンプレート修正なら修正後のテンプレート、設定変更なら設定値を含める]

## 苦労度の基準

| 苦労度 | 意味 |
|--------|------|
| ★☆☆☆☆（1） | 軽微。数分で解決 |
| ★★☆☆☆（2） | やや苦労。調査に少し時間がかかった |
| ★★★☆☆（3） | 中程度。複数回の試行錯誤が必要だった |
| ★★★★☆（4） | かなり苦労。大幅な方針転換が必要だった |
| ★★★★★（5） | 極めて困難。解決に非常に長い時間を要した |

## 深刻度の基準

浪費時間・手戻り規模・再発リスクで判定する。

| 深刻度 | 意味 |
|--------|------|
| ★☆☆☆☆（1） | 軽微。影響は限定的 |
| ★★☆☆☆（2） | 小さい。一部の手戻りで済んだ |
| ★★★☆☆（3） | 中程度。複数ファイルの修正が必要だった |
| ★★★★☆（4） | 大きい。フェーズをまたぐ手戻りが発生した |
| ★★★★★（5） | 致命的。設計の根本的な見直しが必要だった |

## 原因カテゴリ凡例

| カテゴリ | 説明 |
|---------|------|
| ルール不足 | CLAUDE.mdやrules/に必要なルールが記載されていなかった |
| ルール曖昧 | ルールはあるが曖昧で、誤った解釈をした |
| テンプレート不足 | 必要なテンプレートやスキルが用意されていなかった |
| ツール制約 | Claude CodeやMCPツールの制約に起因する問題 |
| 環境問題 | Docker、依存関係、ネットワーク等の環境に起因する問題 |
| context消失 | コンテキストウィンドウの圧縮で情報が失われた |
| 外部依存 | 外部ライブラリやAPIの仕様・不具合に起因する問題 |

## 影響範囲凡例

| 分類 | 説明 |
|------|------|
| gaido側の問題 | gaidoのテンプレート・スキル・Docker設定等に起因する問題 |
| ルール改善で予防可能 | Output System実装時に発生したが、gaido側のルール追加で今後の別プロジェクトでも予防可能 |
| Output System固有 | 特定ライブラリ・API・CSS仕様等に固有。gaido側での対応は不要 |

### 影響範囲の判定基準

| 分類 | 基準 | 例 |
|------|------|-----|
| gaido側の問題 | gaidoのテンプレート、スキル、entrypoint.sh、Docker設定等に起因 | entrypoint.shの設定不足、スキルの動作不良 |
| ルール改善で予防可能 | 原因はOutput System実装時に発生したが、gaidoのrules/constraints.mdにルール追加すれば今後の別プロジェクトでも予防できる | Dockerネットワーク設定のルール不足、mermaid構文の注意事項 |
| Output System固有 | 特定ライブラリ・API・CSS仕様など、今回のOutput System実装に固有。他のプロジェクトでは再現しない、またはルール化のコストが高すぎる | React stale closure、Web Audio API仕様、特定パッケージの互換性 |

## 注意事項

- **推測で書かない**: 実際に経験した事実のみを記載する。「おそらく〜だろう」は不可
- **具体的に書く**: 「環境構築で苦労した」ではなく、何がどう問題だったか具体的に記述する
- **改善提案は即実装可能なレベルで書く**: このproblems.mdを読んだ別のAIエージェントが、追加の調査なしにそのまま実装できること。対象ファイル・変更内容・追記すべきテキストまで具体的に書く
- **最大10項目**: 苦労度・深刻度の高いものを優先して記載する

## 改善提案の書き方（良い例・悪い例）

### 悪い例（抽象的すぎて実装できない）

```markdown
#### 改善提案

| 項目 | 内容 |
|------|------|
| 対象ファイル | `.claude/rules/` 配下 |
| 変更種別 | 新規追加 |

**具体的な変更内容:**

Docker環境でのポート設定に関するルールを追加する。
```

### 良い例（別コンテキストで即実装可能）

```markdown
#### 改善提案

| 項目 | 内容 |
|------|------|
| 対象ファイル | `.claude/rules/docker-rules.md` |
| 変更種別 | 既存修正（末尾に追記） |

**具体的な変更内容:**

以下のルールを追記する:

> ## ポートマッピングの注意事項
>
> Output Systemのコンテナ内部ポートとホスト公開ポートが異なる場合がある。
> docker-compose.ymlの `ports` セクションでは `ホスト:コンテナ` の順で記述する。
>
> - フロントエンドのdevサーバがコンテナ内で5173番で起動する場合でも、ホスト側は3001番で公開する
> - `docker-compose.yml` の `ports` に `"3001:5173"` と記述する
> - アプリケーション側の設定（vite.config.ts等）ではコンテナ内部ポート（5173）を使う
> - Playwrightテストやブラウザからのアクセスではホスト公開ポート（3001）を使う
>
> ```yaml
> # 正しい例
> ports:
>   - "3001:5173"  # ホスト3001 → コンテナ5173
>
> # 間違い例
> ports:
>   - "5173:5173"  # ホスト側も5173にすると他サービスと衝突する可能性
> ```
```
