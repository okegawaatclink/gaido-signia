# 提案書品質チェックリスト

HTMLモック・PPTXの生成後、ユーザーへ提示する前に以下を必ず自己チェックすること。
1つでも NG があれば修正してから提示する。

---

## 【致命的】スライドタイトルの二重表示

スライドタイトルが画面内に2回出現していないか確認する。

よくある原因:
- スライドのヘッダー領域とコンテンツ領域の両方にタイトルを描画している
- draw.io図やPPTX変換時にタイトルが再付与される

**確認方法:** 生成したPNG/HTMLを目視確認し、タイトル文字列が重複していないかチェックする

---

## 【致命的】テーマの統一

全スライドで同一テーマ（ダーク or ライト）を使用しているか確認する。

| NG例 | 修正方針 |
|---|---|
| スライド1〜3はダークテーマ、スライド4以降はライトテーマ | story.mdのデザイン方針で決定したテーマに統一する |
| 同一スライド内でダーク背景とライト背景が混在 | コンテンツエリアの背景色をテーマに合わせて修正する |

---

## 【致命的】プレースホルダの残存

以下のようなダミーテキストが残っていないか全スライドを確認する。

| NG例 | 修正方針 |
|---|---|
| `19XX年`、`X,XXX名`、`XX百万円` | story.mdまたはRFPから実際の値を取得して埋める |
| `○○○○`、`03-XXXX-XXXX`、`xxx@example.com` | 担当者情報をユーザーに確認して埋める（不明な場合は `[要確認]` と明記） |
| `[会社名]`、`[顧客名]`、`TBD` | story.mdの案件情報から補完する |

**確認方法:** 生成したHTMLファイルをgrepでスキャンする
```bash
grep -rn "XX\|○○\|TBD\|\[.*\]\|XXXX\|xxx@" ai_generated/proposals/{案件名}/design/
```

## 【致命的】HTMLシェイプの再現漏れ（PPTX生成時）

HTMLモックに存在するボックス・カード・コネクタ矢印等の図形要素が、PPTXでネイティブシェイプとして再現されているか確認する。

**確認手順:**

1. `design/slide_*.html` の各ファイルを読み込み、以下の要素を列挙する
   - 色付きヘッダー付きボックス（外枠 + ヘッダー帯）
   - 複数列グリッド状に並ぶカード群
   - ボックス間のコネクタ矢印（`→` / `-->` / SVGアロー等）
   - 区切り線・アクセント帯・強調ボックス等の装飾要素

2. `generate_slides.py` の対応するスライド生成コードに `add_shape()` 呼び出しが存在するか確認する

| NG例 | 修正方針 |
|---|---|
| HTMLにボックス群があるのにテキストのみのスライドになっている | 種別を `mixed` に変更し、`add_shape()` でボックスを再現する |
| コネクタ矢印がPPTXに存在しない | `add_shape(RIGHT_ARROW)` またはテキストボックスで `→` を追加する |
| ヘッダー帯付きカードが単なるテキスト箇条書きになっている | 外枠矩形＋ヘッダー矩形を `add_shape()` で生成し、子ボックスを配置する |
| HTMLにあった区切り線・アクセント帯がPPTXに存在しない | `add_shape(RECTANGLE)` で再現する |

**HTMLにない要素を独自に追加してもならない。**

## 情報密度のバラつき

- スライド下半分以上が空白になっていないか確認する
- 空白が目立つ場合は以下のいずれかで対処：
  - 実績・詳細情報を追加して密度を上げる
  - 複数スライドを1枚に統合する（story.mdも更新すること）
- 密度の目安：スライド面積の50%以上をコンテンツが占めること

---

## サブタイトルの崩れ防止（HTML/CSS）

サブタイトルや装飾テキストに取り消し線が入る問題を防ぐため、以下のCSSルールを遵守する。

```css
/* NG: text-decorationを明示指定していないと継承で崩れる場合がある */
.subtitle { font-size: 14px; }

/* OK: 明示的にnoneを指定する */
.subtitle {
  font-size: 14px;
  text-decoration: none;
  text-decoration-line: none;
}
```

また、スライドタイトル・サブタイトルに `<u>` タグや `border-bottom` を装飾目的で使わないこと。

## フォント・表記の統一

| 項目 | ルール |
|---|---|
| 英語技術用語 | 英語で統一（例: `BigQuery`、`ETL`、`API`） |
| 括弧内補足 | 日本語で統一（例: `BigQuery（データウェアハウス）`） |
| 期間表記 | `ヶ月` で統一（`ヶ月` との混在禁止） |
| 波ダッシュ | `〜` で統一 |
| 数字 | 全角数字禁止、半角で統一 |

全スライド横断で同一用語の表記ゆれがないか確認すること。

## 箇条書き記号の統一

| 用途 | 使用記号 |
|---|---|
| 本文の箇条書き | `・`（中黒） |
| 強調箇条書き | `▶`（右向き三角） |
| 評価表（良・可・不可） | `◎` / `○` / `△` |

`●` は評価記号と混同しやすいため本文箇条書きには使用しない。

## カラースキームの凡例

- アーキテクチャ図・ガントチャートなど、色で意味を区別している図には必ず凡例を追加する
- 凡例の形式（図の右下または下部に配置）：

```
凡例: ■ レイヤーA  ■ レイヤーB  ■ レイヤーC
```

- 同一図内で同一レイヤーのボックスは同一色にすること（微妙な色差禁止）

## テキストの重なり防止

要素同士がテキストを重ねて表示していないか確認する。

| NG例 | 修正方針 |
|---|---|
| ラベルテキストがボックスや枠線に重なっている | ラベルの位置・サイズを調整し、隣接要素と重ならない余白を確保する |
| カードや図内のテキストが隣の要素にはみ出している | 要素の幅・高さを広げるか、フォントサイズを縮小する |
| フローやダイアグラム内の補足テキストが図形と重なっている | テキストボックスをノード外に移動するか、ノードサイズを拡張する |

**HTMLの場合:** `overflow: hidden` を指定し、他要素に重ならないようにする。

---

## テーブルの折り返し・カラム幅

テーブルセル内で不自然な箇所での折り返しが発生していないか確認する。

| NG例 | 修正方針 |
|---|---|
| `PostgreSQL` が `PostgreSQ` / `L` と途中で折り返されている | カラム幅を広げ、単語の途中で折り返さない |
| `アプリリ` が `アプリ` / `リ` と1文字だけ次行になっている | カラム幅を広げるか、テキストを短縮する |
| `Spring Boot` が `Spring` / `Boot` と単語の切れ目で折り返されている | カラム幅を調整し、英語の場合は `white-space: nowrap` を検討する |

**確認観点:** セル内の最長テキストが1行で収まるカラム幅を確保すること。
列数が多く幅が足りない場合は、テキストを短縮するか行を分割してテーブルを2段構成にすること。

---

## 費用スライドのレイアウト

- 全項目が `XX百万円` などの未確定値の場合、空欄にせずレンジ（例: `15〜25百万円`）または概算根拠を示す
- 費用内訳が複数ある場合は円グラフや積み上げ棒グラフで配分比率を視覚化する
- 表のカラム幅は内容量に合わせて調整し、空白列を作らない

---

## スケジュールスライドの視認性

- ガントチャートの各バーに開始月・終了月のラベルを明記する
- タイトルと内容の粒度を一致させる：
  - フェーズレベルのチャート → 「フェーズスケジュール」
  - タスクレベルのチャート → 「WBSスケジュール」
- 「WBSベースのスケジュール」と銘打つ場合は、タスク粒度の項目を含めること

---

## 締めスライド（最終ページ）の品質

以下の要素が含まれているか確認する：

- [ ] 「なぜ弊社を選ぶべきか」の訴求ポイント（3点以内で明確に）
- [ ] NEXT STEPS（具体的なアクションと日程）
- [ ] 担当者連絡先（プレースホルダでないこと）
- [ ] レイアウトの空白が全体の30%以内

## 【致命的】フッター表示

**表紙を除く全スライドに以下の2要素を必ず表示すること。どちらか一方でも欠けていれば修正する。**

### フッター領域の配置

```
┌────────────────────────────────────────────────────────────┐
│（スライド本文エリア）                                         │
├────────────────────────────────────────────────────────────┤
│ © ITOCHU Techno-Solutions Corporation    [left]  2 / 33 [right] │
└────────────────────────────────────────────────────────────┘
```

| 要素 | 内容 | 位置 | フォントサイズ |
|------|------|------|-------------|
| コピーライト | `© ITOCHU Techno-Solutions Corporation` | 左下（left: 0.2in） | 9pt、グレー（#808080） |
| ページ番号 | `{現在ページ} / {総ページ数}`（例: `2 / 33`） | 右下（right: 0.2in） | 9〜10pt、グレー系 |
| 表紙（p.1） | 両方とも表示しない | — | — |
| それ以外 | 両方とも全スライドに表示する | — | — |

HTMLで実装する場合の例：

```html
<div style="position: absolute; bottom: 16px; right: 24px; font-size: 12px; color: #666;">
  2 / 33
</div>
```

## 技術選定の根拠

技術スタック・アーキテクチャ選定スライドに以下が含まれているか確認する。

| チェック項目 | NG例 | OK例 |
|---|---|---|
| 採用理由が「定番」「実績あり」だけになっていない | 「Java定番のため」 | 「Java定番かつ貴社他案件でも採用実績があり、スキルトランスファーコストが低いため」 |
| 対抗案との比較が少なくとも1つある | Nuxt.jsのみ記載 | Nuxt.js vs Next.jsの比較観点を明示 |
| 今回の案件課題と選定理由が紐付いている | 汎用的な説明のみ | 「現行課題Xを解決するためにYを採用」 |

## 内容の独自性・具体性

以下のスライドタイプで「どのSIerでも書ける内容」になっていないか確認する。

| スライドタイプ | NG（汎用的） | OK（案件特化） |
|---|---|---|
| 課題理解 | RFPの課題をそのまま箇条書き | RFPの課題＋「CTCとして深掘りした仮説・洞察」を追記 |
| 弊社の強み | 「XX年の実績」のみ | 「XX年の実績」が今回の案件（領域・技術・規模）にどう効くかを明示 |
| 技術適用 | Clean Architectureの一般説明 | 現行システムのどこに適用し、何が改善されるかを具体的に示す |
| 未解決リスク | 「後追い移行を検討」 | 「A案（〇〇）またはB案（〇〇）の2択で判断基準はXX」と方針を提示 |
