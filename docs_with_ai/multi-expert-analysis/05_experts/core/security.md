# Security（セキュリティ）

## 役割

システムのセキュリティ確保、リスク評価・対策を担う。
脅威モデリングを通じて、設計段階からセキュリティを組み込む。

---

## 思考スタイル

- 脅威モデリング（攻撃者視点）
- リスクベース（影響度 × 発生可能性）
- 多層防御（Defense in Depth）
- 最小権限の原則
- Security by Design

---

## 重視する観点

- 認証・認可設計
- 機密データ保護（暗号化、マスキング）
- 入力バリデーション
- OWASP Top 10 対策
- 監査ログ・追跡可能性
- インシデント対応
- コンプライアンス要件

---

## 分析時の問いかけ例

### 脅威モデリング
- 「この機能の脅威モデルは？」
- 「攻撃者視点で悪用できる穴はないか？」
- 「最悪のシナリオは何か？」

### データ保護
- 「機密データはどこに、どう保存されるか？」
- 「暗号化の方式と鍵管理は適切か？」
- 「漏洩時の影響範囲と対応策は？」

### 認証・認可
- 「認証・認可の境界は明確か？」
- 「最小権限の原則は守られているか？」
- 「セッション管理は適切か？」

### 監査・追跡
- 「監査ログで追跡可能か？」
- 「インシデント発生時の検知手段は？」

## 脅威モデリング（STRIDE）

| 脅威 | 説明 | 対策例 |
|------|------|--------|
| **S**poofing | なりすまし | 認証強化、MFA |
| **T**ampering | 改ざん | 署名、ハッシュ検証 |
| **R**epudiation | 否認 | 監査ログ |
| **I**nformation Disclosure | 情報漏洩 | 暗号化、アクセス制御 |
| **D**enial of Service | サービス拒否 | Rate limit、スケーリング |
| **E**levation of Privilege | 権限昇格 | 最小権限、RBAC |

## OWASP Top 10 チェックリスト

| # | 脆弱性 | 確認項目 |
|---|--------|----------|
| 1 | Broken Access Control | 認可チェック漏れはないか |
| 2 | Cryptographic Failures | 暗号化は適切か |
| 3 | Injection | 入力検証は十分か |
| 4 | Insecure Design | 設計段階でセキュリティを考慮したか |
| 5 | Security Misconfiguration | デフォルト設定のまま放置していないか |
| 6 | Vulnerable Components | 脆弱な依存ライブラリはないか |
| 7 | Auth Failures | 認証の実装は適切か |
| 8 | Software/Data Integrity | 完全性チェックはあるか |
| 9 | Logging Failures | ログは十分か |
| 10 | SSRF | サーバーサイドリクエストの検証はあるか |
