# 保守・改修パターン

既存システムの改修・機能追加・リファクタリング向けのパターン。

---

## 概要

| 項目 | 内容 |
|------|------|
| パターン名 | maintenance |
| 用途 | 既存システムの保守・改修 |
| 追加専門家 | なし（core のみ） |

---

## このパターンを選ぶ条件

- [ ] 既存システムがある
- [ ] 後方互換性の考慮が必要
- [ ] 既存コード/データへの影響がある
- [ ] 本番稼働中のシステムに変更を加える

---

## 専門家別の追加観点

### PO

- 既存ユーザーへの影響
- 移行コストの考慮
- ダウンタイムの許容範囲

### Architect

- 影響範囲分析
- 後方互換性の設計
- 技術的負債の扱い（返済 or 許容）
- 移行戦略の選定

### QA

- 回帰テスト重点
- 既存テスト資産の活用
- 本番環境への影響確認

### Security

- 既存の脆弱性チェック
- 新規追加部分のレビュー
- 認証・認可の変更影響

---

## 分析フローの追加ステップ

### Epic分析時

```
1. 標準フロー（INVEST、YAGNI）
   ↓
2. 【追加】既存システム影響分析
   - 影響を受けるコンポーネントは？
   - データマイグレーションは必要か？
   ↓
3. 【追加】移行戦略の決定
   - Strangler / Big Bang / Branch by Abstraction？
   - ロールバック可能か？
```

### PBI分析時

```
1. 標準フロー（INVEST、YAGNI、DoD）
   ↓
2. 【追加】後方互換性チェック
   - 既存APIに影響はないか？
   - 既存データとの整合性は？
   ↓
3. 【追加】回帰テスト範囲の決定
   - どこまでテストするか？
```

---

## 追加チェックリスト

### Epic作成時

- [ ] 影響範囲が特定されているか
- [ ] 後方互換性の方針が決まっているか
- [ ] 移行戦略が明確か
- [ ] ロールバック手順があるか

### PBI作成時

- [ ] 既存機能への影響が分析されているか
- [ ] 回帰テスト範囲が決まっているか
- [ ] データマイグレーション計画があるか（必要な場合）

### Task作成時

- [ ] 既存コードの理解が十分か
- [ ] テスト環境で検証可能か

## 移行戦略

| 戦略 | 概要 | メリット | デメリット | 適用ケース |
|------|------|---------|-----------|-----------|
| **Strangler Pattern** | 新機能をGWで徐々に置換 | 低リスク、段階的 | 時間がかかる | 大規模、継続運用必須 |
| **Big Bang** | 一括で新システムに切替 | 短期間で完了 | 高リスク | 小規模、停止可能 |
| **Branch by Abstraction** | 抽象化層を挟んで分離 | 並行開発可能 | 複雑度増加 | チーム分割、長期移行 |
| **Feature Flag** | フラグで新旧切替 | 即時ロールバック可 | コード複雑化 | A/Bテスト、段階リリース |

## 注意事項

- 「動いているものは触るな」を盲信しない
- ただし、変更の影響範囲は必ず把握する
- 回帰テストなしの変更は禁止
- 本番データを開発環境で使う場合はマスキング必須
