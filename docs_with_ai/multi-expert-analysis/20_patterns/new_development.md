# 新規開発パターン

ゼロからシステム/機能を構築する案件向けのパターン。

---

## 概要

| 項目 | 内容 |
|------|------|
| パターン名 | new_development |
| 用途 | 新規システム・機能の開発 |
| 追加専門家 | なし（core のみ） |

---

## このパターンを選ぶ条件

- [ ] 既存システムがない、または既存に依存しない
- [ ] 技術選定から行う必要がある
- [ ] 白紙から設計可能
- [ ] 後方互換性を考慮しなくてよい

---

## 専門家別の追加観点

### PO

- MVP定義が特に重要
- スコープクリープに注意
- 「あれもこれも」を防ぐYAGNI徹底

### Architect

- 技術選定の根拠を明確に
- 将来拡張性の設計（ただし過剰設計は禁止）
- アーキテクチャ決定記録（ADR）の作成推奨

### QA

- テスト戦略をゼロから構築
- CI/CD パイプライン設計
- 品質基準の初期設定

### Security

- Security by Design
- 脅威モデリングを初期に実施
- セキュリティ要件の早期明確化

## 分析フローの追加ステップ

### Epic分析時

```
1. 標準フロー（INVEST、YAGNI）
   ↓
2. 【追加】技術戦略の確認
   - 技術選定の方向性は決まっているか？
   - 制約となる技術要件はあるか？
   ↓
3. 【追加】MVP範囲の確認
   - 最小限で価値を届けるスコープは？
   - Phase分けは必要か？
```

### PBI分析時

```
1. 標準フロー（INVEST、YAGNI、DoD）
   ↓
2. 【追加】技術選定の詳細化
   - このPBIで使用する技術は？
   - 選定理由と代替案は？
```

## 追加チェックリスト

### Epic作成時

- [ ] 技術選定根拠が明確か
- [ ] MVP範囲が定義されているか
- [ ] 非機能要件が明確か
- [ ] ステークホルダーが特定されているか

### PBI作成時

- [ ] 技術的実現性が確認されているか
- [ ] 依存ライブラリ・サービスが特定されているか
- [ ] セキュリティ観点がレビューされているか

### Task作成時

- [ ] 実装パターンが決まっているか
- [ ] テスト方針が明確か

## 注意事項

- 技術選定は慎重に、ただし決定を先延ばしにしない
- 「後で決める」は技術的負債の温床
- MVPを超えた機能は「後回し」ではなく「バックログ」へ
