ETL/ELT、データパイプライン、分析基盤向けのパターン。

---

## 概要

| 項目 | 内容 |
|------|------|
| パターン名 | data_pipeline |
| 用途 | データ収集・加工・提供基盤の構築 |
| 追加専門家 | **Data Architect（必須）** |

---

## このパターンを選ぶ条件

- [ ] データの収集・加工・提供が主目的
- [ ] Databricks, Snowflake, BigQuery 等を使用
- [ ] データ品質・リネージが重要
- [ ] 分析・BI・MLのためのデータ整備

- レイヤー設計（Bronze/Silver/Gold）

- スキーマ進化
- データリネージ

### QA

- データ品質テスト
- 期待値検証
- 異常検知

### Security

- データ分類（機密度）
- アクセス制御
- 匿名化・マスキング
- コンプライアンス（GDPR等）

## 分析フローの追加ステップ

### Epic分析時

```
1. 標準フロー（INVEST、YAGNI）
   ↓
2. 【追加】データ戦略の確認
   - データソースは何か？
   - データ利用者は誰か？
   - SLA（鮮度、可用性）は？
   ↓
3. 【追加】ガバナンス方針の確認
   - データオーナーは誰か？
   - アクセス制御ポリシーは？
```

### PBI分析時

```
1. 標準フロー（INVEST、YAGNI、DoD）
   ↓
2. 【追加】データモデル設計
   - どのレイヤーに配置するか？
   - スキーマ定義は？
   ↓
3. 【追加】品質ゲートの設定
   - 品質チェック項目は？
   - 異常時の挙動は？
```

## 追加チェックリスト

### Epic作成時

- [ ] データソースが特定されているか
- [ ] データ利用者と用途が明確か
- [ ] SLA（鮮度、可用性）が定義されているか

### PBI作成時

- [ ] データリネージが追跡可能か
- [ ] 品質ゲートが定義されているか

- [ ] データ保持期間・削除ポリシーがあるか

### Task作成時

- [ ] 変換ロジックが明確か
- [ ] 品質チェック実装方針が決まっているか
- [ ] エラーハンドリングが定義されているか

## レイヤー設計（Medallion Architecture）

```
┌─────────────────────────────────────────────────────┐
│  Source                                              │
│  (RDB, API, Files, Streaming...)                    │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  Bronze Layer（生データ取込み）                       │
│  - そのまま保存                                      │
│  - スキーマオンリード                                │
│  - 全履歴保持                                       │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  Silver Layer（クレンジング・標準化）                 │
│  - 型変換、重複排除                                  │
│  - 品質チェック済み                                  │
│  - 統合されたビュー                                  │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  Gold Layer（ビジネス利用）                          │
│  - 集計・結合済み                                    │
│  - BI/ML向け                                        │
│  - ビジネスルール適用済み                            │
└─────────────────────────────────────────────────────┘
```

## データ品質チェック項目

| カテゴリ | チェック項目 | 例 |
|----------|-------------|-----|
| 完全性 | 必須項目がNULLでないか | customer_id IS NOT NULL |
| 一意性 | 重複がないか | DISTINCT count = total count |
| 整合性 | 参照整合性が保たれているか | FK存在チェック |
| 妥当性 | 値が妥当な範囲か | age BETWEEN 0 AND 150 |
| 適時性 | データの鮮度は十分か | updated_at > threshold |
| 正確性 | ソースと一致しているか | record count比較 |

## 処理パターン

| パターン | 特徴 | 適用ケース |
|----------|------|-----------|
| **Batch** | 定期実行、大量処理 | 日次/週次レポート、データ統合 |
| **Streaming** | リアルタイム、低レイテンシ | イベント処理、リアルタイムダッシュボード |
| **Lambda** | Batch + Streaming | 両方の要件がある場合 |
| **Kappa** | Streaming統一 | シンプルさ重視 |

---

## 注意事項

- データリネージは後から追加が困難、最初から設計に含める
- スキーマ変更は「追加」を基本、「変更」「削除」は慎重に
- 本番データの開発利用は匿名化必須
- データ品質問題は早期に検知、後工程に流さない
