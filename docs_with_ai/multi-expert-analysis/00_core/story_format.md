# ユーザーストーリーの書き方

このドキュメントは、Epic/PBI/Task で使用するユーザーストーリーの書き方を定義します。

---

## 1. 基本フォーマット

### 1.1 ユーザーストーリー形式

```
As a [誰が]
I want to [何をしたい]
So that [どんな価値が得られる]
```

日本語の場合：

```
[誰が] として
[何をしたい]
それによって [どんな価値が得られる]
```

### 1.2 タイトルの命名規則

| 階層 | 形式 | 例 |
|------|------|-----|
| Epic | 大きな価値を表現 | 「ユーザーが安全にログインできる」 |
| PBI | 小さな価値を表現 | 「パスワードを忘れても復旧できる」 |
| Task | 動詞で始まる作業 | 「認証APIを実装する」 |

---

## 2. 階層別の書き方

### 2.1 Epic

**目的**: ビジネス価値レベルの大きな目標

```markdown
# Epic: ユーザーが安全にログインできる

## ユーザーストーリー
**As a** 一般ユーザー
**I want to** 安全にシステムにログインしたい
**So that** 自分のデータを保護しながらサービスを利用できる

## ビジネス価値
- セキュリティ要件の充足
- ユーザー信頼の獲得

## 成功指標（KPI）
- ログイン成功率 99%以上
- セキュリティインシデント 0件
```

### 2.2 PBI

**目的**: ユーザーが得られる小さな価値単位

```markdown
# PBI: パスワードを忘れても復旧できる

## ユーザーストーリー
**As a** パスワードを忘れたユーザー
**I want to** パスワードをリセットしたい
**So that** アカウントに再度アクセスできる

## 価値の説明
この PBI が完了すると、ユーザーはパスワードを忘れても
メール経由で自力でパスワードを再設定できる。

## 受入条件
- [ ] リセットメールが5分以内に届く
- [ ] リセットリンクは24時間で無効化される
- [ ] 新パスワードは過去3回と異なること
```

### 2.3 Task

**目的**: 具体的な作業単位（技術タスク含む）

```markdown
# Task: パスワードリセットAPIを実装する

## 親PBI
PBI-001: パスワードを忘れても復旧できる

## 作業内容
- POST /api/password-reset エンドポイント作成
- リセットトークン生成ロジック実装
- メール送信処理実装

## 完了条件
- [ ] エンドポイントが正常に動作する
- [ ] ユニットテストが通る
- [ ] コードレビュー完了
```

## 3. 良い例・悪い例

### 3.1 Epic

**良い例:**
```
Epic: ユーザーが商品を購入できる
Epic: 管理者がレポートを確認できる
Epic: システムが大量アクセスに耐えられる
```

**悪い例:**
```
× Epic: ログイン機能  ← 価値が表現されていない
× Epic: DBテーブル作成  ← 技術作業であり価値ではない
× Epic: Phase1開発  ← 曖昧すぎる
```

### 3.2 PBI

**良い例:**
```
PBI: カートに商品を追加できる
PBI: 注文履歴を確認できる
PBI: クレジットカードで支払える
```

**悪い例:**
```
× PBI: カート機能  ← 「できる」で終わっていない
× PBI: API実装  ← 技術作業、価値ではない
× PBI: バグ修正  ← 具体性がない
```

### 3.3 Task

**良い例:**
```
Task: カートAPIのエンドポイントを実装する
Task: 商品追加のユニットテストを作成する
Task: カート画面のUIコンポーネントを作成する
```

**悪い例:**
```
× Task: カート  ← 動詞がない
× Task: 開発する  ← 何を開発するか不明
× Task: その他作業  ← 曖昧すぎる
```

## 4. ユーザーストーリーのチェックリスト

### 4.1 Epic/PBI共通

- [ ] 「誰が」が明確か（具体的なペルソナ or ロール）
- [ ] 「何をしたい」が行動として表現されているか
- [ ] 「価値」がユーザー視点で書かれているか
- [ ] 技術用語ではなくビジネス用語で書かれているか
- [ ] 1文で理解できるシンプルさか

### 4.2 PBI追加チェック

- [ ] 1スプリントで完了可能なサイズか
- [ ] 受入条件が検証可能な形で書かれているか
- [ ] 他のPBIと独立しているか

### 4.3 Task追加チェック

- [ ] 動詞で始まっているか
- [ ] 作業内容が具体的か
- [ ] 完了条件が明確か
- [ ] 見積もり可能か

## 5. ペルソナの例

ユーザーストーリーの「誰が」に使うペルソナ例：

| カテゴリ | ペルソナ例 |
|----------|-----------|
| エンドユーザー | 一般ユーザー、新規ユーザー、ヘビーユーザー |
| 管理者 | システム管理者、コンテンツ管理者、店舗管理者 |
| ビジネス | 営業担当、カスタマーサポート、経営層 |
| 技術 | 開発者、運用担当、データアナリスト |
| 外部 | パートナー企業、API利用者、監査担当 |
