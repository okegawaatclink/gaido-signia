# 多視点専門家分析フレームワーク

このドキュメントは、どんな案件でも変わらない分析の根幹ルールを定義します。

---

## 1. 基本原則

### 1.1 多視点分析とは

- 複数の専門家視点から議論し、盲点をなくす
- 単一視点のバイアスを排除する
- トレードオフを明示化する
- 合意形成のプロセスを透明化する

### 1.2 専門家の行動規範

- 自分の専門領域から正直に意見する
- 他の専門家の意見を尊重しつつ、異論は明確に述べる
- 根拠なき同調をしない
- 不確実な点は「不確実」と明言する
- 判断の根拠を必ず示す

---

## 2. 分析プロセス

### 2.1 基本ステップ

```
1. コンテキスト理解
   └─ 対象の現状把握、参照ドキュメントの読み込み

2. 専門家招集
   └─ 案件タイプに応じた専門家を決定

3. 多視点議論
   └─ 各専門家が意見を出し、議論・反論・補足

4. 収束・結論
   └─ 合意点・相違点を整理

5. 成果物生成
   └─ 決定事項をアウトプット
```

### 2.2 議論のルール

- 各専門家は独立して意見を述べる
- 反論・補足は根拠を明示する
- 「賛成」「反対」だけでなく「条件付き賛成」もあり
- 全ての議論は同期的に行う（非同期コメントは不可）

---

## 3. 収束条件

### 3.1 終了判定

以下のいずれかで議論を終了：

| 条件 | 説明 |
|------|------|
| 全員合意 | 全専門家が同意 |
| 2回連続同じ結論 | 追加議論で結論が変わらない |
| 指定ラウンド数到達 | 設定されたラウンド数に達した |

### 3.2 結論の形式

| 結論タイプ | 扱い |
|-----------|------|
| **全員合意** | そのまま採用 |
| **多数派意見** | 反対意見も記録して採用 |
| **合意不可** | トレードオフを明示し、判断を上位に委ねる |

---

## 4. 階層別の分析深度

| 階層 | 目的 | 議論の焦点 | 主な成果物 |
|------|------|-----------|-----------|
| **Epic** | 方向性の決定 | Why（なぜやるか） | Epic承認、PBI分解方針 |
| **PBI** | 実現方法の決定 | What（何を作るか） | DoD、受入条件、Task分解 |
| **Task** | 作業内容の確定 | How（どう作るか） | 完了条件 |

---

## 5. 階層別チェック項目

### 5.1 Epic/PBI 共通：INVESTチェック

| 項目 | 確認内容 |
|------|----------|
| **I**ndependent | 他のEpic/PBIと独立しているか |
| **N**egotiable | 交渉・調整の余地があるか |
| **V**aluable | ユーザー/ビジネス価値があるか |
| **E**stimable | 見積もり可能か |
| **S**mall | 適切なサイズか |
| **T**estable | 検証可能か |

### 5.2 Epic/PBI 共通：YAGNIチェック

- [ ] 今この瞬間、これがないと先に進めないか？
- [ ] 作らないことで10倍以上の手戻りが発生するか？
- [ ] 推測ではなく実際の要求に基づいているか？

全て「いいえ」の場合、そのEpic/PBIは後回しまたは削除を検討する。

---

## 6. 成果物の原則

### 6.1 必須要素

全ての成果物には以下を含める：

| 要素 | 説明 |
|------|------|
| 決定事項 | 何が決まったか |
| 根拠 | なぜそう決めたか |
| 残課題 | 未決定・要検討事項 |

### 6.2 トレーサビリティ

- 参照元 → 成果物 → 終了条件 の紐付けを明示
- 参照元の章番号は必ず検証する
- 成果物と終了条件は分離して記載

---

## 7. 専門家の選択

### 7.1 Core専門家（常に参加）

- PO（プロダクトオーナー）
- Architect（アーキテクト）
- QA（品質保証）
- Security（セキュリティ）

### 7.2 Optional専門家（案件に応じて追加）

- Data Architect（データアーキテクト）- data_pipeline パターン時
- （将来拡張可能）

### 7.3 選択方法

`10_project/context.md` で指定：

```yaml
experts:
  core: all
  optional:
    - data_architect
```

---

## 8. 分析の開始条件

分析を開始するには、最低限以下が必要：

| 必須項目 | 説明 |
|----------|------|
| 分析対象 | 何を分析するか（1-2文） |
| 案件タイプ | new_development / maintenance / data_pipeline |

以下は任意（なければAIが質問）：

- プロジェクト背景
- 制約条件
- 参照ドキュメント
