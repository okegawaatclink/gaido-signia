# 移行戦略リファレンス

既存システムから新システムへの移行戦略を選定するためのリファレンス。

---

## 移行戦略一覧

| 戦略 | 概要 | リスク | 期間 |
|------|------|--------|------|
| **Strangler Pattern** | 段階的に置換 | 低 | 長 |
| **Big Bang** | 一括切替 | 高 | 短 |
| **Branch by Abstraction** | 抽象化して分離 | 中 | 中〜長 |
| **Feature Flag** | フラグで新旧切替 | 中 | 中 |
| **Blue-Green** | 環境を並行稼働 | 低〜中 | 中 |
| **Canary** | 段階的にトラフィック移行 | 低 | 中 |

---

## 戦略詳細

### Strangler Pattern（絞め殺しパターン）

```
┌─────────────────────────────────────────┐
│  Gateway / Proxy                         │
│  ┌─────────────────────────────────────┐ │
│  │ /api/v1/users  → 新システム         │ │
│  │ /api/v1/orders → 旧システム         │ │
│  │ /api/v1/...    → 旧システム         │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

| 項目 | 内容 |
|------|------|
| **概要** | Gatewayで新旧を振り分け、徐々に新システムの範囲を拡大 |
| **メリット** | 低リスク、段階的に移行、いつでも戻せる |
| **デメリット** | 時間がかかる、Gateway管理が複雑化 |
| **適用ケース** | 大規模システム、継続稼働必須、長期移行計画 |

---

### Big Bang（一括切替）

```
Time ──────────────────────────────►

旧システム ████████████████████
                              │
                              │ 切替
                              ▼
新システム                     ████████████████████
```

| 項目 | 内容 |
|------|------|
| **概要** | 特定日時に一括で新システムに切り替え |
| **メリット** | 短期間で完了、シンプル |
| **デメリット** | 高リスク、失敗時の影響大、ロールバック困難 |
| **適用ケース** | 小規模、計画停止可能、テスト十分 |

---

### Branch by Abstraction（抽象化分離）

```
┌─────────────────────────────────────────┐
│  Application                             │
│  ┌─────────────────────────────────────┐ │
│  │  Abstraction Layer (Interface)       │ │
│  └─────────────────────────────────────┘ │
│        │                    │            │
│        ▼                    ▼            │
│  ┌───────────┐        ┌───────────┐     │
│  │ Old Impl  │        │ New Impl  │     │
│  └───────────┘        └───────────┘     │
└─────────────────────────────────────────┘
```

| 項目 | 内容 |
|------|------|
| **概要** | 抽象化レイヤーを挟み、新旧実装を切り替え可能に |
| **メリット** | 並行開発可能、テスト容易 |
| **デメリット** | 抽象化による複雑度増加、初期コスト |
| **適用ケース** | チーム分割開発、長期移行、段階リリース |

---

### Feature Flag（機能フラグ）

```
if (featureFlag.isEnabled("new-payment")) {
    newPaymentService.process()
} else {
    oldPaymentService.process()
}
```

| 項目 | 内容 |
|------|------|
| **概要** | フラグで新旧機能を動的に切り替え |
| **メリット** | 即時ロールバック可能、A/Bテスト可能 |
| **デメリット** | コード複雑化、フラグ管理必要、技術的負債 |
| **適用ケース** | 段階リリース、A/Bテスト、ユーザー別切替 |

---

### Blue-Green Deployment

```
┌───────────────────┐     ┌───────────────────┐
│  Blue (Current)   │     │  Green (New)      │
│  ████████████     │     │  ████████████     │
└─────────┬─────────┘     └─────────┬─────────┘
          │                         │
          └─────────┬───────────────┘
                    │
              ┌─────┴─────┐
              │  Router   │ ← 切替
              └───────────┘
```

| 項目 | 内容 |
|------|------|
| **概要** | 2つの本番環境を並行稼働、ルーターで切替 |
| **メリット** | 即時切替、即時ロールバック |
| **デメリット** | インフラコスト2倍、DB同期が課題 |
| **適用ケース** | ダウンタイム禁止、インフラコスト許容 |

---

### Canary Release

```
Traffic Distribution:
  旧システム: 95% ──→ 90% ──→ 50% ──→ 0%
  新システム:  5% ──→ 10% ──→ 50% ──→ 100%
```

| 項目 | 内容 |
|------|------|
| **概要** | 少数ユーザーから段階的にトラフィックを移行 |
| **メリット** | リスク最小化、問題の早期発見 |
| **デメリット** | 監視必須、ロールアウト時間が必要 |
| **適用ケース** | 大規模ユーザー、リスク回避重視 |

---

## 選定フロー

```
ダウンタイム許容できる？
    │
    ├── Yes → Big Bang検討
    │
    └── No
        │
        ▼
    即時ロールバック必要？
        │
        ├── Yes → Blue-Green / Canary
        │
        └── No
            │
            ▼
        段階的に移行したい？
            │
            ├── Yes → Strangler Pattern
            │
            └── No → Feature Flag / Branch by Abstraction
```
