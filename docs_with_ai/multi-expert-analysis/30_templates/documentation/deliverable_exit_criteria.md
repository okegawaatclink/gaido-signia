# 成果物の終了条件

各成果物タイプごとの終了条件（Exit Criteria）を定義する。

---

## 終了条件の目的

- 成果物の完成基準を明確にする
- 品質の一貫性を保つ
- レビュー観点を統一する

---

## 成果物別終了条件

### Epic

| 項目 | 終了条件 |
|------|---------|
| タイトル | ユーザーストーリー形式で記載されている |
| 目的 | ビジネス価値が明記されている |
| スコープ | 含む/含まないが明確 |
| PBI分割 | 子PBIが特定されている |
| INVESTチェック | 全項目確認済み |
| YAGNIチェック | 不要な機能が含まれていないことを確認済み |
| 優先順 | POによる優先順が決定されている |
| コスト概算 | 概算見積もりが記載されている |

### PBI

| 項目 | 終了条件 |
|------|---------|
| タイトル | ユーザーストーリー形式で記載されている |
| 受入条件 | Given-When-Then形式で記載されている |
| 完成の定義（DoD） | QAにより定義されている |
| Task分割 | 実装可能なTaskに分解されている |
| INVESTチェック | 全項目確認済み |
| YAGNIチェック | 不要な機能が含まれていないことを確認済み |
| 見積もり | 相対見積もりが完了している |
| 依存関係 | 他PBIとの依存が特定されている |

### Task

| 項目 | 終了条件 |
|------|---------|
| タイトル | 作業内容が明確 |
| 作業内容 | 具体的な手順が記載されている |
| 完了条件 | 何をもって完了とするか明記されている |
| 見積もり | 時間見積もりが記載されている |
| 担当者 | アサイン可能な状態 |

---

## 分析成果物の終了条件

### 設計ドキュメント

| 項目 | 終了条件 |
|------|---------|
| アーキテクチャ図 | 主要コンポーネントと接続が図示されている |
| 技術選定理由 | 選定根拠が記載されている |
| 非機能要件 | 性能・可用性・セキュリティ要件が定義されている |
| 縮退設計 | 外部サービス障害時の動作が定義されている |

### セキュリティ分析

| 項目 | 終了条件 |
|------|---------|
| 脅威分析 | STRIDE分析が完了している |
| リスク評価 | 各脅威のリスクレベルが評価されている |
| 対策 | 各リスクに対する対策が定義されている |

### 品質計画

| 項目 | 終了条件 |
|------|---------|
| テスト戦略 | テストレベルごとの方針が定義されている |
| カバレッジ目標 | 数値目標が設定されている |
| 回帰テスト | 回帰テスト対象が特定されている |

## レビュー観点

### POレビュー観点

- ビジネス価値が妥当か
- 優先順は適切か
- スコープは適切か
- YAGNIの観点で不要なものがないか

### Architectレビュー観点

- 技術的に実現可能か
- 非機能要件を満たせるか
- 既存システムとの整合性があるか

### QAレビュー観点

- テスト可能か
- 受入条件は検証可能か
- エッジケースが考慮されているか

### Securityレビュー観点

- セキュリティリスクが特定されているか
- 対策は十分か
- コンプライアンス要件を満たすか
