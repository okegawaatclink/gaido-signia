# 回帰テスト判定フローガイドライン

回帰テストの要否を判定するためのフローチャート。

---

## 判定フロー

```
タスク/受入条件
    │
    ▼
[必須] タグか？ ─────────── Yes ──→ 回帰テスト ○（必須）
    │
    No
    │
    ▼
過去に不具合があった箇所か？ ─── Yes ──→ 回帰テスト ○（必須）
    │
    No
    │
    ▼
変更頻度が高い機能か？ ───────── Yes ──→ 回帰テスト ○（必須）
    │
    No
    │
    ▼
外部システム連携部分か？ ─────── Yes ──→ 回帰テスト ○（必須）
    │
    No
    │
    ▼
自動化コストが低いか？ ───────── Yes ──→ 回帰テスト △（推奨）
    │
    No
    │
    ▼
回帰テスト ×（手動 or 対象外）
```

---

## 判定基準の詳細

### ○ 必須（自動回帰テスト必須）

| 条件 | 理由 |
|------|------|
| `[必須]` タグ | 当たり前品質、欠けると致命的 |
| 過去に不具合 | 再発リスクが高い |
| 変更頻度が高い | 影響を受けやすい |
| 外部システム連携 | 変更の影響を受けやすい |

### △ 推奨（状況に応じて回帰テスト）

| 条件 | 理由 |
|------|------|
| 自動化コストが低い | 投資対効果が高い |
| `[推奨]` タグ | あった方が安心 |

### × 対象外（手動 or 回帰テスト不要）

| 条件 | 理由 |
|------|------|
| `[任意]` タグ | なくても問題ない |
| 一度きりの処理 | 再実行されない |
| UIの見た目のみ | 機能に影響しない |

## 記入例

```markdown
## 回帰テスト判定記録：認証機能

| # | 機能/条件 | [必須]? | 過去不具合? | 変更頻度? | 外部連携? | 自動化コスト | 判定 |
|---|----------|---------|------------|----------|----------|-------------|------|

| 3 | パスワードリセット | Yes | No | 低 | Yes(SMTP) | 中 | ○ |
| 4 | ログイン画面の表示 | No | No | 低 | No | 中 | △ |
| 5 | ダークモード対応 | No | No | 低 | No | 高 | × |
```

---

## 自動化の優先順位

```
1. ○ 必須 かつ 自動化コスト低 → 最優先で自動化
2. ○ 必須 かつ 自動化コスト高 → 工夫して自動化
3. △ 推奨 かつ 自動化コスト低 → 余裕があれば自動化
4. △ 推奨 かつ 自動化コスト高 → 手動でカバー
5. × 対象外 → 自動化しない
```
