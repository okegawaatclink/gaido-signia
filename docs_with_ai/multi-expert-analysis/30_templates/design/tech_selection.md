# 技術選定根拠テンプレート

技術選定の理由と却下した代替案を記録するためのテンプレート。

---

## 使い方

1. カテゴリごとに選定技術を記載
2. 選定理由を明記
3. 検討した代替案と却下理由を記録
4. 将来の見直し基準を定義

---

---

## 評価観点

| 観点 | 確認項目 |
|------|----------|
| 実現性 | 要件を満たせるか |
| 成熟度 | 本番実績はあるか |
| コミュニティ | サポート・情報は十分か |
| 学習コスト | チームが習得可能か |
| 運用性 | 監視・デバッグは容易か |
| コスト | ライセンス・インフラ費用 |
| 将来性 | 開発が継続されているか |

## 記入例

```markdown
## 技術選定根拠：Webアプリケーション

| カテゴリ | 選定技術 | 選定理由 | 却下した代替案 |
|---------|---------|---------|---------------|
| 言語 | TypeScript | 型安全、JS互換、エコシステム | JavaScript: 型がない |
| フレームワーク | Next.js | SSR/SSG、App Router、Vercel連携 | Nuxt: チームスキル、React: 機能不足 |
| ORM | Prisma | 型安全、マイグレーション、DX | TypeORM: DX劣る、Drizzle: 新しすぎる |
| DB | PostgreSQL | 信頼性、機能豊富、RLS | MySQL: JSON機能弱い、SQLite: スケール不可 |
| Linter | Biome | 高速、ESLint+Prettier相当 | ESLint: 設定複雑、遅い |
| テスト | Vitest | 高速、TS対応、Jest互換 | Jest: 遅い |
| CI/CD | GitHub Actions | GitHub統合、YAML定義 | CircleCI: 追加コスト |
| インフラ | Vercel | Next.js最適化、デプロイ容易 | AWS: 運用コスト高、Netlify: SSR弱い |
```

## 見直し基準

```markdown
## 見直しトリガー

以下の場合、技術選定を見直す：

- [ ] 選定技術のEOL（End of Life）が発表された
- [ ] 重大なセキュリティ脆弱性が発見され、パッチが提供されない
- [ ] パフォーマンス要件を満たせなくなった
- [ ] ライセンス変更により使用が制限された
- [ ] チームスキルの変化で運用困難になった
```
