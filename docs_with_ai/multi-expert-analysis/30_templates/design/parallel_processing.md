## 記入例

```markdown
## 並列処理設計：バッチデータ処理

### 並列処理パラメータ

| 項目 | 設定値 | 根拠 |
|------|--------|------|
| 同時実行数 | 5並列 | 外部API rate limit (100 req/min) 考慮 |
| バッチサイズ | 100件/バッチ | メモリ使用量とAPI制限のバランス |
| タイムアウト | 30000ms | 処理時間の実測値(10s) × 安全係数(3) |
| リトライ回数 | 3回 | 一時的障害からの復旧 |
| リトライ間隔（初回） | 1000ms | API推奨値 |
| バックオフ倍率 | 2倍 | 指数バックオフ（1s→2s→4s） |

### トークン/サイズ上限管理

| 項目 | 上限値 | 超過時の対応 |
|------|--------|-------------|
| 単一ファイル | 10MB | スキップ、警告ログ |
| 単一リクエストトークン | 4000トークン | 分割して逐次処理 |
| バッチ合計トークン | 100000トークン | 自動分割 |
| 1日あたりリクエスト | 10000回 | 処理停止、翌日再開 |

### レート制限対応

| API | Rate Limit | 対応策 |
|-----|-----------|--------|
| OpenAI API | 60 req/min (tier1) | 同時実行5、間隔1s |
| 社内API | 100 req/min | 同時実行10、間隔600ms |
| 外部データソース | 1000 req/hour | キュー管理、分散実行 |

### エラーハンドリング

| エラー種別 | 対応 |
|-----------|------|
| タイムアウト | リトライ（最大3回） |
| 429 (Rate Limit) | 待機（Retry-After ヘッダー参照）+ リトライ |
| 5xx (サーバーエラー) | リトライ（指数バックオフ） |
| 400 (不正リクエスト) | スキップ、エラーログ |
| 401/403 (認証エラー) | 処理停止、管理者通知 |
| ネットワークエラー | リトライ（最大3回） |
```

## 並列処理パターン

### パターン1: 固定並列度

```
┌─────────────────────────────────┐
│  入力キュー                      │
│  [1][2][3][4][5][6][7][8]...    │
└─────────────┬───────────────────┘
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
┌───────┐ ┌───────┐ ┌───────┐
│Worker1│ │Worker2│ │Worker3│  ← 並列度3
└───────┘ └───────┘ └───────┘
```
